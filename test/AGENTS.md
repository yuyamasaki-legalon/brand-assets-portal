# AI Agent Guidelines

This file contains core instructions for AI agents working on this project. For detailed information, reference the comprehensive documentation in `docs/`.

## Guardrails

### Security
- **NEVER commit secrets**: Do not commit `.env` files, API keys, tokens, passwords, or credentials
- **NEVER expose sensitive data**: Do not log or expose user data, credentials, or internal configurations
- **NEVER bypass authentication**: All protected routes and API endpoints must properly validate authentication

### Code Quality
- Use `src/pages/template/` as the code reference — sandbox (`src/pages/sandbox/`) contains experimental prototypes that may not follow standards or best practices
- Use Aegis design system components (`@legalforce/aegis-react`) instead of creating custom UI — this keeps visual consistency across the product and reduces maintenance burden
- Fix all TypeScript errors before considering work complete — strict mode is enabled, so type errors usually indicate real problems
- Do not modify `dist/` or `node_modules/` — these are regenerated by build/install and any manual changes will be overwritten

### Layout Implementation
- Consult `src/pages/template/` before implementing any page layout — the Layouts/Pages/States templates (list, settings, detail, fill, form, chat, dashboard, etc.) encode tested spacing, responsive, and composition patterns that are easy to get wrong from scratch

### Concept Usage
- When working on service-specific pages (`loc/`, `dealon/`, `workon/`), read `src/pages/template/{service}/CONCEPT.md` before implementation — concept documents define domain terminology, and skipping them leads to inconsistent labels and variable names
- Match prompt keywords against feature CONCEPT.md `keywords` to load relevant domain knowledge

### Workflow
- **NEVER push directly to main**: Create a feature branch and submit a PR — direct pushes are irreversible and bypass review
- **NEVER deploy without approval**: Deployments require explicit team authorization — unauthorized deploys can affect production users
- Run `pnpm format && pnpm fix:style` before committing — CI will reject unformatted code, so catching issues locally saves roundtrips
- Run `pnpm build` before opening a PR — this catches type errors and build failures before reviewers spend time on the PR
- Follow the Prototype Workflow (see below) to ensure requirements are captured automatically
- To pin a stable preview URL for a project, add a `preview:{slug}` label to the PR (see `docs/workflow-guide.md` → Preview URL Configuration)

## Sandbox Page Implementation

Before writing ANY sandbox page code, follow these steps in order:

1. **Identify layout category**: Read `.claude/rules/template-layouts.md` to determine the layout type (list, detail, settings, chat, form, dashboard, etc.)
2. **Search template catalog**: Search `src/pages/template/CATALOG.md` for keywords matching the user's request — find the closest existing template
3. **Read matching template source**: Open and read the template file identified in step 2. This is the starting point, not a blank file
4. **Scaffold and implement**: Run `pnpm sandbox:create` to create the page (this handles sandbox registration in `index.tsx` / `routes.tsx`), then implement based on the template code from step 3

See `docs/rules/ui/09_layout_composition.md` for the build order (PageLayout frame → internal layout → variant/spacing).

## Concept Hierarchy

Domain knowledge is managed through a 3-tier concept document hierarchy (co-located with code):

```
Service Concept   → src/pages/template/{service}/CONCEPT.md
  └─ Feature Concept → src/pages/template/{service}/{feature}/CONCEPT.md
      └─ Prototype Concept → PRD.md / SPEC.md (existing)
```

See `.claude/rules/concept-hierarchy.md` for details. Use `/concept-manager` skill for concept management.

## Prototype Workflow

When building sandbox UI from user prompts, follow this workflow to ensure requirements are captured automatically:

1. **Create sandbox page** — Use `pnpm sandbox:create` for page creation and sandbox registration
2. **Auto-generate PRD** — Automatically create/update `auto-generated-prd.md` from the user's prompt. Do NOT ask the user to fill in requirements — infer them yourself and classify as Confirmed vs Inferred
3. **Iterate prototype with PRD tracking** — As the user refines requirements, keep `auto-generated-prd.md` updated in the background. No user action needed
4. **Auto-generate Handoff** — Automatically create/update `auto-generated-handoff.md` by analyzing prototype code. Extracts component usage, data models, API contract hints, and production code mappings for engineer handoff. No user action needed
5. **Auto-generate Insights** (conditional) — If iteration signals are detected (corrections, re-requests, direction changes), automatically create/update `auto-generated-insights.md`. Skip this step when the session has no iteration signals. No user action needed

**Critical**: PRD and Handoff generation are the AI's responsibility. Insights generation is conditional — only when iteration signals are detected. Never ask the user to write or review these documents. Extract requirements from conversation context, analyze code for handoff details, and keep files up to date automatically.
