# チーム・メンバー機能（追加機能）実装要求ドキュメント（AI駆動開発用）

---

## 1. 背景/目的（ユーザー価値）

参照（上位要求）: `documents/analytics-dashboard-requirements.md`

この機能（チーム・メンバー分析）は、法務部門のマネージャー/担当者が以下を達成することを支援する。

- 負荷分散: 誰が忙しいかを把握し、アサイン調整の判断材料を得ながら、案件の担当者をアサインする
- 遅延防止: 納期逼迫・超過、滞留を早期に発見する
- 育成/評価: 個人の経験（案件タイプ/契約類型）と成果（処理件数・リードタイム等）を把握する

### 1.1. 対象ユーザー（本PRD内の定義）

- 本PRD内で「ユーザー」と言う場合、マターマネジメントのProライセンスを持つユーザーを指す
  - 本ページのチーム集計・表示対象となる「メンバー」の母集団は、このProライセンス保有ユーザーである
  - 表示・集計対象の案件は 6.1 の認可/権限制約（閲覧可能な案件のみ）に従う

---

## 2. 主要ユースケース

### 2.1. ユースケースA: 今の危険信号を見つけ、対応したい（モニタリング）

- チーム全体の進行中案件を、納期の近さ/超過でざっくり把握する
- チームメンバーの現在の状況（進行中件数、納期逼迫/超過、滞留など）や、担当してきた案件タイプの傾向を参考にしながら、未割り当て案件の主担当をアサインする
- 納期超過/逼迫・滞留が多いメンバー、または該当案件へドリルダウンして詳細を確認できる

### 2.2. ユースケースB: 仕事量の偏りを見たい（負荷分散）

- メンバー別に、担当案件数（主/副）の内訳を比較する
- どのステータス/案件タイプが負荷の原因かを把握する

### 2.3. ユースケースC: 過去の傾向を見たい（分析・評価）

- 月次の新規/完了の推移、リードタイム、初回返信などの指標を確認する
- 個人の指標を、期間・案件タイプ等で絞り込める

## 3. 画面仕様

この章では「画面で何ができるか」を、実装・テスト可能な粒度で定義する。

### 3.1. 画面構成

- メインタブ
  - `案件`（モニタリング/負荷分散、未割り当てのアサイン）
  - `パフォーマンス`（傾向/評価）
- ドリルダウン
  - メンバーを選ぶと、該当メンバーの詳細（進行中案件一覧 + 指標）を表示できる

### 3.1.1. **【未決】**画面状態（表示設定/フィルタ）の保存方針

この画面は、テーブルの表示設定（カラムの移動/表示非表示/幅調整など）や、各種フィルタ/表示切替が多い。
どこまで可能にするか、どこまで保存するかを決めないと実装がぶれるため、ここで方針を決める。

保存先の候補:
- 保存しない（ページ再読み込みでリセット）
- ローカルストレージに保存（ユーザーごと）
- データベースに保存（テナントごと）

決める項目:
| 対象 | できること（UI） | 保存する？ | 保存先 | 備考 |
| --- | --- | --- | --- | --- |
| 未割り当て一覧（テーブル） | カラム移動 / 表示非表示 / 幅調整 / 固定 | **【未決】** | **【未決】** | 固定は「担当者/副担当者を右端固定」は別要件として必須 |
| メンバー詳細Pane（案件一覧） | カラム移動 / 表示非表示 / 幅調整 / 固定 / ソート | **【未決】** | **【未決】** | 主担当一覧/副担当一覧の両方に適用するかも含めて決める |
| チームの案件状況（メンバー別内訳） | フィルタ条件 / 表示軸 / 表示形式（グラフ/テーブル） / 案件数ソート | **【未決】** | **【未決】** |  |
| パフォーマンス分析 | フィルタ条件 / 表示項目ON/OFF / 表示形式（グラフ/テーブル） | **【未決】** | **【未決】** |  |
| リードタイム構成分析 | フィルタ条件 / 表示モード切替 / 表示形式（グラフ/テーブル） / ステータス分類設定 | **【未決】** | **【未決】** | ステータス分類設定は「テナント設定」としてDB保存（3.3 B-2） |

### 3.1.2. アサイン保存のUX

- 前提: サービス本体には別の「案件一覧」ページがあり、保存時の表示/挙動は原則としてそのページの動きに準拠する
- 保存タイミング: 変更時に即保存
- 権限:
  - 6.1 の認可/権限制約に従う（更新権限がない場合は入力できない/保存できない）
- 保存結果の基本:
  - 成功通知（トースト等）: 出さない
  - 一覧から除外されるなどの画面変化がある場合は、Bの各仕様に従う
  - 失敗時の表示/挙動は、案件一覧ページの動きに準拠する

- 準拠事項:
  - 失敗時の入力値の扱い（ロールバック/値維持など）
  - 再試行導線（自動再試行、手動再試行ボタン等）
  - 競合（同時編集）時の通知
  - 保存中の連続変更の扱い
  - エラー種別ごとの扱い（権限/バリデーション/ネットワーク/サーバー等）
  - 監査/履歴（変更履歴や監査ログ）

### 3.1.3. ナビゲーション（案件詳細への遷移）

- 対象: 案件が並ぶ「案件一覧テーブル」（未割り当て一覧、メンバー詳細Paneの案件一覧、その他案件行を持つ一覧）
- 遷移:
  - テーブルの行をクリックすると、該当の案件詳細ページに遷移する
  - 別タブで開く動線も提供する（例: 行内のリンク/アイコン、またはコンテキストメニュー）
- 同一タブ遷移時の画面状態:
  - 同一タブで案件詳細へ遷移する場合、遷移前の本画面の状態（フィルタ条件/表示切替/表示設定など）をローカルストレージへ保存し、戻ったときに復元できるようにする
    - 保存する項目は 3.1.1 の検討対象と同じ（少なくともフィルタ条件と表示切替は復元対象）

### 3.2. 案件タブ

#### A. 概要（何を見るタブか）

- チーム全体の“進行中”の案件を中心に、納期・滞留・内訳を見て、個人へドリルダウンする

#### B. 表示項目（最低限）
- 迷ったら「表示対象」「集計単位」「操作」「例外（空/権限/欠損）」の順で書く。

##### B-1. 未割り当ての案件

主担当が空の案件を一覧し、主担当/副担当のアサインを行う。

- 主担当が空（未割り当て）
  - 対象ステータス: ステータスが完了のものは除く
  - 未割り当て判定: `assigneeUserId` が `null` の案件
  - 表示件数: 最大10件（先頭10件）を表示する
  - 先頭10件の定義: `updatedAt` が新しいものが上になるように並べ、上位10件を表示する
  - 一覧で確認できる
  - 表示列:
    - 案件名 / 納期 / 依頼者 / 依頼部署 / 担当者（主担当：入力可能）
    - （後述の「副担当者が空」ONの場合）副担当者（入力可能）
  - 一覧の行クリック:
    - 行をクリックすると、該当の案件詳細ページに遷移する（3.1.3 に従う）
  - 一覧上で「案件担当者（主担当）」をアサインできる
    - 選択後、画面に即時反映される
    - アサイン結果はサービス本体へ保存され、再表示しても反映される
    - 保存が成功したら、当該案件は「未割り当ての案件」一覧から除外（表示されなくなる）する
  - 競合（同時編集）の扱い: 後勝ち（最後に保存した担当者で上書き）

- 副担当の出し分け（副担当者が空の案件を表示）
  - 「副担当者が空の案件を表示」のチェックをONにすると:
    - 表示対象の定義:
      - `assigneeUserId` が `null` または `subAssigneeUserId` が `null` の案件に絞り込まれる
    - 「副担当者」列が追加され、一覧上で副担当者も選択できる
      - 副担当者の選択結果はサービス本体へ保存され、再表示しても反映される
      - 保存が成功したら、当該案件は（副担当が空ではなくなるため）この一覧から除外（表示されなくなる）する
  - 表示件数: 最大10件（先頭10件）を表示する
  - 先頭10件の定義: `updatedAt` が新しいものが上になるように並べ、上位10件を表示する

##### B-2. チームの案件状況

- 納期サマリー（フィルター）
  - 対象: 進行中（7.1 の定義に従う）
  - 基準日（今日）: テナントの設定（US/JP）に従う（日付境界/タイムゾーン）
  - 納期（`dueAt`）: 日付まで（時刻は持たない）
  - 区分: `すべて / 納期超過 / 今日まで / 〜3日後 / 〜1週間後 / 1週間後〜 / 納期未入力`
  - **【未決】**区分の境界（日単位）:
    - 以下は現時点の案（例）
    - 納期超過: `dueAt` が `基準日（今日）` より前
    - 今日まで: `dueAt` が `基準日（今日）` と同日（=0日）
    - 〜3日後: `dueAt` が `基準日（今日）+1〜3日`
    - 〜1週間後: `dueAt` が `基準日（今日）+4〜7日`
    - 1週間後〜: `dueAt` が `基準日（今日）+8日以上`
  - 納期未入力: 案件に付与されている納期（`dueAt`）が空
  - 表示: 各区分の件数を表示（対象は進行中）
  - クリック時の挙動:
    - クリックした区分でフィルターし、一覧に反映される
      - 影響範囲: チームの案件状況（メンバー別内訳）だけでなく、メンバー詳細Paneの案件一覧にも影響させる
    - 解除: 「すべて」カードのみ（同じ区分を再クリックしても解除しない）
    - 初期フィルタ: 「すべて」
    - **【未決】**URL: エンジニアに要相談
    - **【未決】**状態保存: 3.1.1 の方針に従う
    
- メンバー別の案件内訳
  - フィルター: 案件タイプ / ステータス / 集計対象（主担当のみ / 副担当のみ / 主+副）
    - 集計対象の初期値: 主担当のみ
  - 表示軸: `ステータス別 / 案件タイプ別 / 滞留状況別`
  - 表示形式: `グラフ` / `テーブル`
  - グラフのインタラクション:
    - マウスオーバー: 積み上げ棒の各エリアにマウスオーバーすると、該当の案件名が表示される
    - クリック: 積み上げ棒の各エリアをクリックすると、そのエリアに対応する条件でフィルターされた状態でメンバー詳細にドリルダウンする
  - 集計単位: メンバーが「案件担当者（主担当）」に設定されている案件数と、「副担当者」に設定されている案件数
  - ステータス別: テナントに設定されているステータス名をそのまま出す
  - 案件タイプ別: サービスで規定している案件タイプをそのまま出す
  - 滞留状況別:
    - 滞留 = ステータスに変更がない状態が何日続いているか
    - 前提: 案件作成時点で必ず何かしらのステータスが付与される（ステータス履歴が空になることは想定しない）
    - 滞留日数の起点: 最後にステータスが変更された日時 もしくは 作成日時
    - 滞留状況の区分（バケット）: 0-3日 / 4-7日 / 8-14日 / 15日以上
  - 主/副の扱い: 両方表示時に重複カウントを許可する
  - ソート:
    - ソートキー: 表示中の案件の総件数
      - 主担当のみ: 主担当として持っている案件の総件数
      - 副担当のみ: 副担当として持っている案件の総件数
      - 主+副: 上記の合算値
    - デフォルト: （候補: 少ないほうが上）
    - **【未決】**同率時: （候補: ユーザー名の昇順。明示しない場合はデータ取得順に依存）


#### D. ドリルダウン（メンバー詳細・案件）

- 表示: 画面右のPaneに「メンバー詳細」を表示する
  - Pane内にも `案件` / `パフォーマンス` のタブがある
  - 初期表示は、ドリルダウン元に応じてタブ/フィルタ条件が引き継がれる
- 対象: 進行中（7.1 の定義に従う）
- 一覧:
  - 主担当としての一覧 / 副担当としての一覧を分けて表示する
  - 並び: 納期が空 → 納期が近い順
  - フィルタ: ステータス / 案件タイプ / 納期区分
  - 表示設定（カラム移動/表示非表示/幅調整/固定/ソート、保存の有無）: 3.1.1 の方針に従う
  - 行クリック: 行をクリックすると、該当の案件詳細ページに遷移する（3.1.3 に従う）
- 表示項目（最小）:
  - 共通: 案件名 / ステータス / 納期 / 受付日（作成日） / 滞留日数（算出できる場合）
  - 主担当一覧: 上記に加えて、副担当者名を最後に表示する
  - 副担当一覧: 上記に加えて、案件担当者名（主担当者名）を最後に表示する
- 権限: 6.1 の認可/権限制約に従う（閲覧可能な案件のみ表示、更新権限がある場合のみ編集可能）

### 3.3. パフォーマンスタブ

#### A. 概要（何を見るタブか）

- 月次での件数/指標推移を確認し、必要に応じて個人へドリルダウンする

#### B. チーム全体の推移（最低限）

- 上部入力（タブ共通）:
  - メンバー（複数選択）
    - 初期状態: すべてのユーザー（1.1 の定義に従う）
    - 集計への反映有無: 反映する
    - 反映範囲:
      - B-1 / B-2 の表示・集計対象に反映する
      - B-2 の「メンバー別」表示（縦軸/一覧）の対象メンバーは、選択したメンバーの集合に限定する
      - B-1 の集計対象案件は、選択したメンバー（主担当）に紐づく案件に限定する
    - 空選択の扱い: すべてのユーザーとして扱う（=フィルタ無し）
      - メンバーがフィルタされていない状態（空選択）のときは、案件担当者（主担当）が未割り当ての案件も集計に含む
  - 集計期間（開始日/終了日）
    - 集計への反映有無: 反映する
    - 反映範囲: パフォーマンスタブ全体（B-1 / B-2 / C）の表示・集計対象に反映する
    - 入力形式: 「YYYY年M月 〜 YYYY年M月」
    - 初期値: 直近6ヶ月
  - リセット: メンバー/期間の入力をクリアする

##### B-1. カード: パフォーマンス分析

- 目的: チーム全体の月次推移を俯瞰する
- 入力UI:
  - 表示項目: 表示する系列をON/OFFできる
    - 新規案件数
    - 完了案件数（内訳: 納期内完了 / 納期超過完了 / 納期未入力）
    - リードタイム（中央値）
    - 初回返信速度（中央値）
  - フィルター: 案件タイプ
- 表示:
  - 表示形式: グラフビュー / テーブルビュー
    - 初期表示: グラフビュー
  - 複合グラフで表示する（案件数: 棒、日数系: 線）
  - 凡例/ツールチップを表示する
- 操作:
  - 表示項目のON/OFFは、即時にグラフへ反映する
  - フィルターを開閉できる
  - 表示項目/フィルター等の画面状態の保存: 3.1.1 の方針に従う

- 集計/表示ルール（確定）:
  - 欠損/0件の表示:
    - 対象期間に該当データがない月の扱い: 0で表示する
    - 中央値が算出できない場合の表示: ハイフン（-）で表示する
  - ソート/表示順: 月の並び（時系列の昇順: 古い月 → 新しい月）

- **【未決】**決めること（実装がブレやすい項目）:
  - **【未決】**テーブルビューの仕様（B-1）:
    - 列（何を出すか）/ 行（何が1行か）/ 並び（ソート）/ 集計単位（月次の粒度・内訳の出し方）を定義する
  - 指標の定義/期間フィルタの基準日時/未決事項は 7.5 に集約（7.5参照）
  - **【未決】**クリック時の挙動:
    - 凡例クリック以外に、棒/線/点をクリックしたときに絞り込み/ドリルダウンを行うか

##### B-2. カード: メンバー別 リードタイム構成分析
- 目的: メンバー別に、リードタイムや案件数、初回返信の偏りを把握する

- 入力UI:
  - 案件タイプ
  - 集計対象（主担当・副担当 / 主担当のみ / 副担当のみ）
    - 注記: 構成分析では主担当固定（本カード内の「集計/表示ルール（確定）」に従う）。案件数・初回返信の表示側に適用する
  - 表示切替: 構成分析 / 案件数・初回返信
  - ステータス分類設定:
    - 構成分析の表示時のみ表示し、分類設定ダイアログを開ける

- 表示:
  - 表示形式: グラフビュー / テーブルビュー
    - 初期表示: グラフビュー
  - 構成分析:
    - メンバー別に、ステータス分類ごとの代表値（日数）を積み上げ棒で表示する
    - 軸: 横軸=日数 / 縦軸=メンバー
    - 表示（ラベル/強調）:
      - 納期超過の案件があるメンバーであっても、メンバーラベルを赤くする等の強調は行わない
  - 案件数・初回返信:
    - メンバー別に、完了案件数を「納期内完了/納期超過完了/納期未入力」の積み上げ棒で表示し、初回返信速度（中央値）を点で重ねて表示する
    - 軸: 案件数=件 / 初回返信速度=日
  - 凡例/ツールチップを表示する

- 操作:
  - メンバー名のクリックで、メンバー詳細Paneへドリルダウンできる
  - **【未決】**棒/点（系列）クリック時の挙動（フィルタとして反映するか、Paneに引き継ぐ条件）
  - 並び替え（ソート）:
    - 推奨（初期値）:
      - 構成分析: 「構成分析の合計（積み上げ合計日数）の多い順」
      - 案件数・初回返信: 「完了案件数（積み上げ合計件数）の多い順」
    - 同率時: ユーザー名の昇順

- 集計/表示ルール（確定）:

## 7. 指標・集計の定義

ここは「正解がサービス本体にある」前提なので、プロトタイプ実装の計算は参考に留める。

### 7.1. 進行中/完了の定義

- 完了:
  - 案件ステータスが `完了` の案件
  - 完了日時は「ステータス履歴で最後に完了になった日時」を採用する（完了→別ステータスへ戻る可能性があるため）
- 進行中:
  - 案件ステータスが `未着手` と `完了` 以外

### 7.2. 納期超過の定義

- 納期: 案件に対して設定されている日付（`dueAt`）
- “超過”の判定: 日単位

#### 7.2.1. 日単位の判定ルール

- 比較対象:
  - `dueAt`（納期）
  - `baseDate`（基準日。画面表示時の「今日」）
- 判定:
  - `baseDate` の日付が `dueAt` の日付を過ぎている場合（`baseDate` > `dueAt`）を「超過」とする
  - 同一日（`baseDate` = `dueAt`）の場合は「超過」ではない
  - 基準時刻/タイムゾーン: 日付境界の扱いはテナント設定に従う

### 7.3. 滞留の定義

- 滞留 = ステータスに変化がない状態
- 滞留日数の起点:
  - `CaseStatusHistory` の「最後にステータスが変化した日時」から今日まで
- バケット: 0-3日 / 4-7日 / 8-14日 / 15日以上

### 7.4. リードタイム/初回返信

- リードタイム:
  - 集計対象: 現在の案件ステータスが「完了」の案件に限る
  - 起点: `createdAt`（案件作成日時）
  - 終点: `CaseStatusHistory` の「最後に完了になった日時」
- 初回返信:
  - 起点: `createdAt`（案件作成日時）
  - **【未決】**終点:
    - 候補a: 案件の依頼者（`requesterUserId`）に対して「返信メッセージ」が送られた最初の時点（`CaseMessage`ベース）
    - 候補b: 未着手→他ステータスへの最初の遷移（`CaseStatusHistory`ベース）

- 休日/営業時間補正:
  - 今回は補正しない（休日/営業時間を除外しない）
  - 将来的に、営業日/営業時間を除外した指標（営業時間ベース）を採用するかは検討する（5.4）

### 7.5. 月次集計の定義

- 集計単位: 月（`YYYY-MM`）
- 集計単位（月次）の境界: テナント設定に従う（タイムゾーン/日付境界）
- 集計期間フィルタの基準日時（指標ごと）:
  - 新規案件数: 案件作成日時（`createdAt`）
  - 完了案件数: 完了した日時（「最後にステータスが完了になった日時」）
  - リードタイム: 完了した日時（「最後にステータスが完了になった日時」）
  - 初回返信: 初回返信日時（7.4 の定義に従う）
- 新規件数:
  - `createdAt` の月でカウント
  - 案件ステータスに関係なく、すべての案件を対象とする
- 完了件数:
  - 定義:
    - 候補: 現在の案件ステータスが完了の案件のうち、「最後にステータスが完了になった日時」が対象月に入る案件数
    - 内訳の扱い:
      - 「納期未入力（`dueAt` が空）」の完了案件も内訳としてカウントする
      - 「納期内完了/納期超過完了」の判定は日付単位で行う（7.2 の日単位ルールに従う）
- 代表値:
  - リードタイム/初回返信速度: 中央値
  - **【未決】**構成分析の代表値: 平均/中央値（B-2 の方針に従う）
- **【未決】**中央値の計算対象:
  - リードタイム/初回返信速度の中央値は、該当月に完了（または初回返信が発生）した案件のみを対象にする、でよいか
  - **【未決】**初回返信が存在しない案件の扱い
    - 候補: 中央値の計算からは除外し、別途「初回返信なし件数」を表示する（またはツールチップで補足する）
