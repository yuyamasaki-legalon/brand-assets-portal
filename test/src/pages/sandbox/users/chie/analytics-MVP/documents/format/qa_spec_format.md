### 4.6 シナリオ記述ルール（推奨）

シナリオ仕様書の一貫性・保守性を高めるための指針。個別仕様書では本節を参照し、重複を避ける。

**記述の基本**

- 1ステップは1つの検証対象にする。Given/When/Thenの粒度は「E2E自動化時に1ステップへ落とせる」程度
- 日本語で統一する（文体・語尾は同じパターンに揃える）
- タグで「優先度 / 種別 / 対象層 / 依存機能」を表現する（例: `@priority:high`, `@type:smoke`, `@layer:ui`）
- 各 Gherkin ブロックには開始 ```gherkin と終了 ``` を必ずペアで書く。見出しごとにブロックを分ける場合は開始忘れに注意する

**用語・表現**

- **サービス上に規定されている用語を用いる**: 仕様にはサービス上で定義・区別されている用語だけを使う。規定されていない用語（ペルソナ用の役割名・ロール名など）は使わない。制限（例: 特定ライセンス保有者に限定）は共通前提・表示条件で別途明示する
- **正式な定義の所在**: シナリオ仕様書単体で理解できるようにする。他ドキュメントにしかない用語・設定（集計対象の定義、表示条件など）は、シナリオ仕様書内の「用語・定義」や「共通前提」にも正式定義を記載する
- **データの抽象化**: 具体的な数値や名前は使わず、抽象表現（N件、0件、1件以上、多数、特定のメンバー、選択中のメンバーなど）を使う。Examples の件数も「0件」「1件以上」「多数」で意図を表す
- **振る舞い中心・ドメイン用語による期待値**: 操作(How)ではなく振る舞い(What)を記述する（例: 「〇〇を選択する」「〇〇が識別できる」）。デザイン詳細（色・アイコン種別）は書かず、ユーザーが得る情報や状態（「視覚的に強調される」「選択中であることが識別できる」等）で記述する

**Scenario Outline・Background**

- **Scenario Outline の活用**: 入力値で結果が変わるだけのパターンは Scenario Outline と Examples に集約する。条件が変わるだけの複数シナリオは1本にまとめ、受け入れ基準も1行に集約する。Examples にエッジケース（0件・多数件・データなし等）を1〜2行含める
- **有限な選択肢の扱い**: 取り得る値が有限で列挙可能な設定（例: 集計対象の3通り）は、用語・定義または共通前提に**一括で1か所だけ**定義する。シナリオではプレースホルダー（`<担当種別>` 等）と Scenario Outline の Examples で参照し、本文・Rule・ステップ中で毎回すべての値を列挙しない
- **Background**: ブロック内で繰り返す前提（タブ表示など）は Background にまとめる。一方、ユーザーが画面上で変更できる設定（集計対象、フィルター、表示軸など）は初期値・固定値として Background に書かず、必要なシナリオの Given で明示する。初期値は Rule で定義してよい

**検証可能性**

- **表示条件・アクセス条件**: ページの表示条件（メニュー表示有無、閲覧可否など）は共通前提に記載し、かつシナリオ（Scenario Outline 等）でも検証可能に書く
