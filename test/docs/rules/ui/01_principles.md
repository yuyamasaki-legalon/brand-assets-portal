# デザイン原則

この章では、LegalOn Technologiesが提供するプロダクトをデザインする上での「憲法」となる、普遍的な考え方や価値観を定義します。

> 【公式ドキュメント】
> 
> - [LegalOn Technologies デザイン原則](https://www.notion.so/LegalOn-Technologies-80c90960e8504b5bab6f4c28c42f92ed?pvs=21)

これらの原則は互いに連携し合っています。「明快」なUIは「予測的」な体験の土台となり、「連続的」な設計は「明快」さを担保します。

**原則の活用方法:**
これらの原則は、デザインレビュー時の評価基準や、新しいUIの方向性で迷った際の判断基準として活用してください。常に「このデザインは3つの原則を満たしているか？」と自問することで、プロダクトの品質を高めることができます。

## 原則1: 明快（Clear）

**余分な情報を削ぎ落とす。すべての人に、負担なく伝わるデザインを。**

### なぜ重要か？

- **専門性と平易性の両立:** 私たちのプロダクトは、専門的な内容を扱いながらも、スキルレベルに関わらず「誰でもわかる・誰でも使える」ことが求められます。明快なUIは、たまにしか利用しないユーザーでも迷わず使える、シンプルな体験を提供するための基本です。
- **認知負荷の軽減:** 複雑な情報や多くの選択肢は、ユーザーの認知的な負担を増やし、タスクの達成を妨げます。UIから不要な要素を削ぎ落とすことで、ユーザーは最も重要な情報やアクションに集中でき、より効率的に業務を進めることができます。
- **アクセシビリティの向上:** 明快でシンプルなUIは、認知特性の多様性を持つ人々を含む、より多くのユーザーがプロダクトを利用するための基本原則です。
  - **【参考文献】** [Making Content Usable for People with Cognitive and Learning Disabilities (W3C)](https://www.w3.org/TR/coga-usable/)

### 実践のための指針

- **【ルール】** 1つの画面の主目的は1つに絞り、その達成を妨げる要素は配置しません。
  - **（理由）** ユーザーが最も重要なアクションに集中でき、より効率的に業務を進めることができるようにするためです。
  - **【補足】**
    - **悪い例:** ユーザー設定画面で、プロフィール情報と通知設定と支払い方法が混在している。
    - **良い例:** 各設定項目を明確に分離し、ユーザーが単一の目的に集中できるようにする。
- **【ルール】** UI上の言葉は、専門用語を避け、可能な限り平易なものを選びます。
  - **（理由）** 専門用語は一部のユーザーの理解を妨げ、「誰でも使える」というプロダクトの基本原則を損なうためです。
  - **【補足】** 専門用語の使用は最小限に留め、必要な場合はツールチップなどで補足情報を加えます。
- **【ルール】** UIは、ヘルプページに頼らずとも、それ自体で完結して理解できるように設計します。
  - **（理由）** ヘルプページを読まなければ使えないUIは、設計が複雑すぎるか、情報が不明瞭である証拠です。これはユーザーのタスクフローを中断させ、認知負荷を高めます。
- **【ルール】** 情報は、ユーザーが必要とするタイミングで段階的に開示します（プログレッシブ・ディスクロージャー）。
  - **（理由）** 一度にすべての情報を見せると、ユーザーは圧倒され、認知負荷が高まるためです。
  - **【補足】** 「詳細設定」などのUIパターンを活用し、初心者から専門家まで、すべてのユーザーが自分に必要な情報量を選択できるようにします。
- **【参考文献】**
  - [Aesthetic and Minimalist Design (Usability Heuristic #8)](https://www.nngroup.com/articles/aesthetic-minimalist-design/)
  - [Recognition Rather Than Recall (Usability Heuristic #6)](https://www.nngroup.com/articles/recognition-not-recall/)
  - [Progressive Disclosure](https://www.nngroup.com/articles/progressive-disclosure/)

## 原則2: 連続的（Sequential）

**共通の表現を取り入れ、連続性を保つ。すべての人に、信頼されるデザインを。**

### なぜ重要か？

- **学習コストの削減:** サービス間でUIの見た目や操作方法が一貫していると、ユーザーは新しい機能を学ぶ際にも、これまでの経験を活かすことができます。これは、私たちのプロダクトが今後サービス間連携を強化していく上で、極めて重要な要素です。
- **品質と信頼性の担保:** 一貫したUIは、プロダクトが細部まで注意深く設計されていることの証であり、ユーザーからの信頼につながります。
- **開発効率の向上:** 共通のデザインシステムを利用することで、同じようなUIを何度もデザイン・開発する必要がなくなり、チームはより本質的な課題解決に集中できます。
- **アクセシビリティと堅牢性の向上:** テスト済みの共通コンポーネントを利用することで、様々なブラウザや支援技術での動作が保証され、プロダクトの堅牢性（Robustness）が高まります。
    - **【参考文献】** [Understanding the Four Principles of Accessibility (W3C)](https://www.w3.org/TR/UNDERSTANDING-WCAG20/intro.html#understanding-the-four-principles-of-accessibility)

### 実践のための指針

- **【ルール】** 顧客向けのプロダクト開発では、必ずデザインシステム「Aegis」を利用します。
  - **（理由）** Aegisで定義された色、タイポグラフィ、コンポーネントを正しく利用することが、プロダクト全体の一貫性を担保する最も確実な方法だからです。
> 【公式ドキュメント】
> - [2025年度LOTエンジニアリング戦略](https://www.notion.so/2025-LOT-19131669571280f18c6ec9954bb2a87b?pvs=21)
> - [Aegis Guidebook - Aegisとは何か](https://www.notion.so/Aegis-Our-Design-System-Aegis-Guidebook-For-Everyone-Involved-in-Product-Development-1d13166957128030a5dccc556d279c4c?pvs=21)
- **【ルール】** 標準的なUIパターンから逸脱しません。
  - **（理由）** ユーザーは他の多くのWebサイトで慣れ親しんだデザインパターンと同じように、あなたのプロダクトが機能することを期待します（ヤコブの法則）。
  - **【補足】** Aegisの標準から逸脱したり、新しいUIパターンを検討したりする際の必須要件として、以下のすべてを満たす必要があります。
    1.  新しいパターンが、既存のどのAegisコンポーネントよりも**明らかに優れている**ことを、客観的なデータで証明できる。
        - **【判断基準】** 「優れている」とは、以下のうち少なくとも1つ以上を、ユーザビリティテストなどの客観的な手法で実証できることを指します。
          - **効率性:** ユーザーが主要なタスクをより速く完了できる。
          - **正確性:** ユーザーが操作を誤る頻度がより低い。
          - **主観的満足度:** ユーザーがより高い満足度を感じる。
    2.  アクセシビリティの国際基準であるWCAGのレベルAAを満たすことが保証されている。
  - **【参考文献】**
    - [Jakob's Law of Internet User Experience](https://www.nngroup.com/articles/jakobs-law-internet-ux/)
    - [ウェブコンテンツ・アクセシビリティ・ガイドライン (WCAG) 2.1 解説書](https://waic.jp/docs/WCAG21/Understanding/)

## 原則3: 予測的（Predictive）

**相手の次の行動を考え、解決に導く。すべての人に、驚きと感動を与えるデザインを。**

### なぜ重要か？

- **認知・操作負荷の最小化:** 私たちが作るサービスは多機能になりがちです。システムがユーザーの目的を予測し、次に行うべきアクションを提示したり、操作を自動化したりすることで、ユーザーの負担を最小限に抑える必要があります。また、専門業務においては、ヒューマンエラーを未然に防ぎ、操作の確実性を高めることも不可欠です。
- **予測による「真の」業務効率化:** 単に手作業をデジタル化するだけでなく、ユーザーの思考を先回りすることで、本来不要であったはずのステップそのものをなくします。これにより、ユーザーはより本質的で創造的な業務に集中でき、真の業務効率化が実現します。
- **AIとの自然な協調:** LLMなどを活用したAI機能が、ユーザーの業務に自然に溶け込むためには、この「予測的」な思想が不可欠です。ユーザーが意識することなくAIの能力を引き出し、その恩恵を受けられるようなインターフェースを目指します。

### 実践のための指針

- **【ルール】** ユーザーが最も少ない手数で目的を達成できる体験を設計します。
  - **（理由）** ユーザーの真の目的を理解し、その達成までの最短経路を提供することが、予測的デザインの核心だからです。
  - **【補足】** クリック数、画面遷移、テキスト入力、繰り返し操作などを極力なくすことを目指します。
- **【ルール】** 賢い初期値を用意します（スマートデフォルト）。
  - **（理由）** ユーザーが何かを入力・設定する際に、最も選ばれる可能性が高いであろう選択肢をあらかじめ設定しておくことで、ユーザーは何も考えずに、あるいは最小限の修正でタスクを完了できます。
- **【ルール】** AIによる提案や分析結果を表示する際は、その根拠や理由を明快に示します。
  - **（理由）** 専門家が業務でAIを使いこなすには、その提案を信頼できることが不可欠です。AIの思考プロセスを透明化することで、ユーザーは結果を鵜呑みにするのではなく、納得して活用できます。
- **【参考文献】**
    - [Anticipatory Design: How to Create Magical User Experiences](https://www.nngroup.com/articles/anticipatory-design/)
    - [Error Prevention (Usability Heuristic #5)](https://www.nngroup.com/articles/preventing-user-errors/)
    - [Smart Defaults](https://www.nngroup.com/articles/smart-defaults/)

## 原則4: オブジェクト指向（Object-Oriented）

> NOTE: ここで定義するオブジェクト指向UI（OOUX）は、LegalOn Technologies全体の統一されたデザイン原則ではなく、本UIガイドライン独自の設計思想です。

* **【指針】** 私たちのプロダクトは、オブジェクト指向UI（Object-Oriented User Interface）の考え方を設計の根幹とします。
    * **（理由）** ユーザーは現実世界と同じように、まず目的の「モノ（オブジェクト）」を見つけ、次にそれに対して「何をするか（アクション）」を考えます。この自然な思考プロセスにUIを合致させることで、学習しやすく、直感的に操作できるインターフェースを実現するためです。
    * **【補足】** 例えば、ユーザーは「契約書を編集する」と考えます。UIも同様に、まず「契約書」の一覧を提供し、ユーザーが特定の契約書を選択した後に、「編集する」というアクションを提供すべきです。この原則に基づき、UIは常にオブジェクトを中心に構成されます。
* **【参考文献】** [OOUI – オブジェクトベースのUIモデリング](https://www.sociomedia.co.jp/7279)
