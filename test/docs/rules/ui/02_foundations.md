# Foundations（UIの基礎要素）

この章では、UIを構成する最小単位の要素（色、文字、余白など）の考え方や使い方を定義します。これらのFoundationsは、第1章で定義したデザイン原則を、実際のUIとして具現化するための基本的なルールセットです。

# 2.1 Color（色）

色はUIの印象を決定づけ、専門的な情報が持つ意味（危険度や重要度など）を直感的に伝える重要な要素です。

> 【公式ドキュメント】
> - [Colorのお話（より詳細な解説）](https://www.notion.so/Color-181316695712809c8a61d56c577e229e?pvs=21)
> - [Aegis Color Token 一覧 (Storybook)](https://main--634618c98b322242afd2ae10.chromatic.com/?path=/docs/tokens-color--docs)

### 色の原則

- **【ルール】** 色の定義は、デザインシステム「Aegis」で定義されたColor Tokenに従います。
    - **（理由）** プロダクト全体で一貫した色の使用法を徹底し、ブランドイメージを統一するためです。また、Tokenを利用することで、将来的なダークモード対応やブランドカラーの変更に、効率的に対応できます。
    - **【補足】** AegisのComponentに設定されている色は、原則として変更しません。独自の色を使いたい場合は、必ずAegisで定義されたColor Tokenから選択してください。
- **【ルール】** 色だけで情報を伝えてはいけません。
    - **（理由）** 色の識別が困難なユーザーが、重要な情報を見落としてしまうことを防ぐためです。これはアクセシビリティの必須要件（WCAG 1.4.1）です。
    - **【補足】** 状態の変化やエラーなどを伝える際は、色に加えて、アイコンやテキスト、UIの形状の変化など、他の視覚的な要素を必ず併用します。
- **【ルール】** テキストと背景のコントラスト比は、WCAGのレベルAAの基準（`4.5:1`）を満します。
    - **（理由）** 弱視や高齢のユーザーでも、テキストを無理なく読むことができるようにするためです。
    - **【補足】** AegisのComponentは、このコントラスト比を満たすように設計されているため、原則としてデフォルトの色設定を変更しないでください。

### 色の役割

- **Neutral Colors（中立色）**
    - **【定義】** UIの背景、枠線、テキストなど、インターフェースの基本的な骨格を構成する色（黒、白、グレー）。
    - **【補足】** Aegisでは、プロダクトのブランドカラーを加味したNeutralカラーを機械的に作成できます。
- **Functional Colors（機能色）**
    - **【定義】** 特定の意味を持つ、アクションや状態を示すための色。
        - **Information（青）:** お知らせや情報提供
        - **Success（緑）:** 操作の成功
        - **Warning（黄）:** 注意喚起
        - **Danger（赤）:** エラーや破壊的な操作の警告
    - **【ルール】** これらの色は、定義された意味以外の目的で使用してはいけません。
        - （補足） 【悪い例】エラーや警告ではないにもかかわらず、特定の機能を目立たせるためだけに`Danger`（赤）を使用する。
- **Accent Colors（アクセントカラー）**
    - **【定義】** ユーザーの注意を引きつけたり、情報を分類したりするために使用する、意味を持たない装飾的な色。
    - **【ルール】** アクセントカラーに特定の意味（例: A社は青、B社は緑）を持たせてはいけません。
        - **（理由）** 人間が識別できる色の数には限りがあり、すぐに色が足りなくなってしまうためです。また、色覚の多様性を持つユーザーには、その意味が伝わらない可能性があります。

# 2.2 Typography（文字）

文字は、専門的な情報を正確に、そして分かりやすく伝えるための最も重要な要素です。

> 【公式ドキュメント】
> - [Aegis Typography Token 一覧 (Storybook)](https://main--634618c98b322242afd2ae10.chromatic.com/?path=/docs/tokens-typography--docs)

### フォントファミリー

- **【ルール】** フォントは、ユーザーのOSに設定されているデフォルトフォントを優先的に使用します。
    - **（理由）** ユーザーが最も見慣れているフォントで表示することで、可読性を高め、認知負荷を下げることができます。
    - **【補足】** 欧文フォントは、専門的な文書で多用される数字や記号の視認性が高く、長時間の作業でも疲れにくい特性を持つ「Inter」を引き続き使用します。

### テキストスタイル

- **【ルール】** テキストは、その情報が持つ意味と構造に合ったスタイル（Title, H1, H2, Body, Captionなど）を適用します。
    - **（理由）** 適切なHTMLタグ（`<h1>`, `<h2>`, `<p>`など）と紐付けることで、見た目の整理だけでなく、スクリーンリーダーの利用者がページ全体の構造を把握し、効率的に読み進めるために不可欠だからです。これはアクセシビリティの必須要件です。

### フォントサイズ

- **【ルール】** 本文テキストの基本サイズは`14px` (`medium`)とします。
    - **（理由）** 専門家が長時間、複雑な文章を読み書きする業務においては、可読性を担保し、眼精疲労を軽減することが不可欠だからです。
    - **【補足】** `12px` (`small`)は補足情報に限定します。
- **【ルール】** `10px`未満のフォントサイズは使用を禁止します。
    - **（理由）** 一部のブラウザでは、ユーザー設定によって最小フォントサイズが定められており、そもそも表示できない、あるいは意図せず表示が崩れる可能性があるためです。
- **【参考文献】** [Legibility, Readability, and Comprehension: Making Users Read Your Words](https://www.nngroup.com/articles/legibility-readability-comprehension/)

# 2.3 Spacing（余白）

余白は、複雑な情報を整理し、専門家が長時間利用しても疲れない、認知負荷の低い画面を作るための重要なデザイン要素です。

> 【公式ドキュメント】
> - [Aegis Spacing Token 一覧 (Storybook)](https://main--634618c98b322242afd2ae10.chromatic.com/?path=/docs/tokens-spacing--docs)
- **【ルール】** 余白の定義は、Aegisで定義されたSpacing Token（`4px`, `8px`, `16px`など）に従います。
    - **（理由）** 一貫した余白のルールを適用することで、画面全体に秩序と安定感が生まれ、ユーザーは無意識のうちに情報のまとまりを理解できます。
- **【ルール】** コンポーネントにプリセットされた余白（`margin`, `padding`）を独自に上書きすることは原則として禁止します。
    - **（理由）** 各コンポーネントは、それ自体で完結し、他の要素と組み合わせた際のリズムが考慮されています。個別の調整はUI全体の一貫性を損ない、システム全体の破綻に繋がります。
    - **【補足】** ページ全体のレイアウトにおける余白は、Aegisが提供するサンプルページを参考にしてください。
- **【指針】** 余白を恐れず、情報を詰め込みすぎないようにします。
    - **（理由）** 関連する要素を近づけ、関連しない要素を離すことで、情報のグループを視覚的に伝えてください。これにより、ユーザーは無意識のうちに情報を整理でき、複雑な専門業務における認知負荷を大幅に削減できます。
- **【参考文献】** [Visual Hierarchy: Organizing Content to Follow Natural Eye Movement](https://www.nngroup.com/articles/visual-hierarchy-ux-definition/)

# 2.4 Iconography（アイコン）

アイコンは、機能や情報を瞬時に伝え、UIの操作性を向上させるための視覚言語です。

> 【公式ドキュメント】
> - [Iconのお話（より詳細な解説）](https://www.notion.so/Icon-18131669571280a6abefc04b34ab7d39?pvs=21)
> - [Aegis Icon一覧 (Chromatic)](https://main--634618c98b322242afd2ae10.chromatic.com/?path=/docs/components-icon--docs)
- **【ルール】** アイコンだけで情報を伝えてはいけません。
    - **（理由）** アイコンの解釈はユーザーに依存するため、意図の誤解が重大なミスに繋がりかねない専門業務ツールにおいては、特に避けるべきだからです。また、スクリーンリーダーの利用者には、アイコンの形や色だけではその意味が伝わりません。
    - **【補足】** デザイナーは、スクリーンリーダーで読み上げられるべき代替テキスト（`aria-label`など）をFigma上で明確に注釈し、その意図を開発者に伝える責任があります。
- **【指針】** アイコンのみで構成されたボタン（アイコンボタン）の使用は、UIの理解をユーザーの解釈に委ねてしまうため、極力避けるべきです。可能な限り、アイコンにはテキストラベルを併記した、より明確なボタン形式を採用します。
    - **【補足】** ただし、「削除（ゴミ箱アイコン）」や「閉じる（×アイコン）」など、OSレベルで意味が広く認知されており、誤解の余地が極めて少ないアイコンについては、この限りではありません。
- **【ルール】** やむを得ずアイコンボタンを使用する場合は、必ずツールチップを表示し、そのボタンが持つ機能を明確に説明しなければなりません。
    - **（理由）** アイコンの意味を補足し、全てのユーザーが機能を正確に理解できるようにするためです。これはアクセシビリティ（WCAG 4.1.2）の要件を満たす上で不可欠です。
- **【ルール】** アイコンは、抽象的で普遍的な形を選びます。
    - **（理由）** 具体的すぎるアイコンは、特定の状況でしか意味が通じず、汎用性が低くなるためです。また、細かく作り込まれたアイコンは、小さく表示されるUI上では識別が困難になります。
    - **【補足】** 新しいアイコンを作成する前に、必ずAegisに既存のアイコンがないか確認してください。一般的な操作（保存、削除、編集など）には、広く認知されている標準的なアイコンを使用します。
- **【参考文献】** [Icon Usability](https://www.nngroup.com/articles/icon-usability/)

# 2.5 Surfaces（サーフェイス）

サーフェイスは、UI要素の「表面」が持つ視覚的な特性（枠線、角丸、影）を定義します。これらは、UI要素の境界を明確にし、階層構造を視覚的に伝え、操作可能な要素を識別しやすくする役割を担います。

> 【公式ドキュメント】
> - [Aegis Border Token 一覧 (Storybook)](https://main--634618c98b322242afd2ae10.chromatic.com/?path=/docs/tokens-borders--docs)
> - [Aegis Corner Radius Token 一覧 (Storybook)](https://main--634618c98b322242afd2ae10.chromatic.com/?path=/docs/tokens-radius--docs)
> - [Aegis Shadow Token 一覧 (Storybook)](https://main--634618c98b322242afd2ae10.chromatic.com/?path=/docs/tokens-shadows--docs)

- **【ルール】** UI要素の枠線、角丸、影のスタイルは、すべてAegisで定義されたデザイントークンに従います。
    - **（理由）** UI全体の視覚的な一貫性を保ち、ユーザーが要素の役割や階層を直感的に理解できるようにするためです。トークンを利用することで、デザインの変更や調整を効率的かつ正確に行うことができます。

### Borders（枠線）
- **【指針】** 枠線は、コンポーネントの物理的な境界を示すというよりも、情報の意味的な境界を明確にするために使用します。
    - **（理由）** 線を多用すると画面の情報量が増え、UIが窮屈な印象になります。特に、Aegisではアクセシビリティ要件を満たすために入力フォームなどの境界が明確にされており、意図的にコントラストも高く設定されているため、必要以上の枠線は視覚的なノイズになりがちです。情報のグルーピングは、まず余白で行うことを検討してください。

### Corner Radius（角丸）
- **【ルール】** Aegisコンポーネントにプリセットされた角丸（`Corner Radius`）を独自に上書きすること、およびトークンとして定義されていない角丸を使用することは原則として禁止します。
    - **（理由）** 角丸はUI全体の印象やトーンを決定づける重要な要素です。一貫性のない角丸が混在すると、UIの品質が低下し、ユーザーにまとまりのない印象を与えてしまうためです。
- **【指針】** 角丸の大きさは、要素のサイズや重要度と関連付けます。例えば、カードのような大きなコンテナは大きな角丸を、ボタンやタグのような小さな要素は小さな角丸を適用することで、視覚的な階層が生まれます。

### Shadows / Elevation（影）
- **【ルール】** Shadowは、UI要素のz軸方向の高さ（Elevation）を表現し、要素が一時的に他のコンテンツの前面に表示されていることを示す場合にのみ使用します。装飾目的での使用は原則として禁止します。
    - **（理由）** ShadowはUIにz軸の概念をもたらし、ユーザーは無意識のうちに画面の階層構造を理解します。単なる装飾として使用すると、この一貫した構造が崩れ、どの要素が操作可能で、どの情報が一時的なものなのかというメンタルモデルを混乱させる原因となるためです。
    - **【補足】** `Modal`、`Dropdown`、`Tooltip`など、ユーザーのアクションによって一時的に他のコンテンツの前面に表示されるコンポーネントに適用されます。【悪い例】静的なカード要素を他と区別したり、目立たせたりするためだけに影を適用する。
