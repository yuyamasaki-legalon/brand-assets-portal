# アクセシビリティ

この章では、プロダクトのアクセシビリティを担保するための基本方針と、その思想的背景となるWCAGの4原則を定義します。

# 7.1 基本方針と現在の取り組み

* **【指針】** LegalOn Technologies開発チームは、JIS X 8341-3:2016の達成等級AAに配慮し、WCAG 2.1、WCAG 2.2に準拠することを目指します。
    * **（理由）** 年齢や身体能力、利用環境に関わらず、すべての人が私たちのプロダクトに平等にアクセスできる環境を構築するためです。
    * > 【公式ドキュメント】 [LegalOn Technologies 開発チーム アクセシビリティ方針](https://www.notion.so/legalforce/LegalOn-Technologies-eb5ea18fddf1423498c4d351ed056234)

* **【指針】** デザインシステム「Aegis」を正しく利用することで、アクセシビリティの基本要件が担保される状態を目指します。
    * **（理由）** Aegisの各コンポーネントは、リリース基準としてアクセシビリティ要件（コントラスト比、キーボード操作、スクリーンリーダー対応など）を満たすように設計・テストされています。
    * **【補足】** デザインや実装に関してアクセシビリティ対応に自信がない場合は、Aegisが提供している標準的なオプションの範囲内でUIを構築することを強く推奨します。
    * 【参考文献】 [Aegisカラーパレットの設計（note）](https://note.com/lf_designers/n/n3fa1689044f7)

> NOTE: 現在、Aegisコンポーネントからアクセシビリティのチェックを開始しており、順次プロダクト全体へと対象を拡大していく予定です。

# 7.2 WCAGが定める4つの原則

このセクションでは、プロダクトが準拠を目指すWCAGの4つの基本原則を定義します。これは、すべてのUIを設計・実装する上での基本的な考え方となります。個別のコンポーネントにおける具体的な実装ルールは、デザインシステム「Aegis」の各ガイドラインで定義します。

*   **知覚可能 (Perceivable)**
    *   **【定義】** 情報やUIコンポーネントは、ユーザーが知覚できる方法で提示されなければなりません。
    *   **【補足】** 例えば、色覚に依存しない情報伝達（アイコンやテキストの併用）や、十分なコントラスト比の確保などがこの原則に含まれます。

*   **操作可能 (Operable)**
    *   **【定義】** UIコンポーネントやナビゲーションは、ユーザーが操作可能でなければなりません。
    *   **【補足】** 例えば、すべての機能がキーボードで利用できることや、現在どこを操作しているかが視覚的に明確であることなどがこの原則に含まれます。

*   **理解可能 (Understandable)**
    *   **【定義】** 情報およびUIの操作は、ユーザーが理解可能でなければなりません。
    *   **【補足】** 例えば、ナビゲーションの仕組みをプロダクト全体で一貫させ、ユーザーの学習コストを下げることなどがこの原則に含まれます。

*   **堅牢 (Robust)**
    *   **【定義】** コンテンツは、支援技術を含む様々なユーザーエージェントが確実に解釈できるほど堅牢でなければなりません。
    *   **【補足】** 例えば、スクリーンリーダーなどの支援技術がUIの役割や状態を正しく解釈できるよう、WAI-ARIAの仕様に沿って適切に実装することなどがこの原則に含まれます。
